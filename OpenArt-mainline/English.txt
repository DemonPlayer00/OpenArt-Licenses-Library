Article 1 – Definitions

1.1 “Work” means any intellectual creation in the literary, artistic, or scientific domain that possesses originality and can be expressed in a tangible form, including but not limited to images, music, sound, video, text, digital models, and their source files (original creation files).

Scope of this License:

    Recommended scope: artistic images (including raster, vector, digital paintings, photographic works, etc.), multimedia artworks (including animation, video, interactive media), and their corresponding project files (source project files, layer files, 3D model source files, etc.).

    Minimum scope: all finished images (final image files intended for display, regardless of format).

    Covered form: any file transmitted over a network or stored offline falls under the scope of this License.

1.2 “Trust Network” means the mutual commitment relationship established among all artists and organizations that use, distribute, or receive Works under this License. Within this Network, all parties commit not to handle or disseminate Works in any way that would cause impairment to the Work’s Core Content Data or Color Configuration.

1.3 “Unmodified Artwork” means a Work that satisfies both of the following conditions:

(a) Core Content Data strictly identical: for images – the pixel matrix; for audio – the sample sequence; for video – the frame sequence; for 3D models – vertex/texture data; for other formats – the essential creative content data – must be bit‑identical to the state when completed by the author. Lossless compression encodings are permitted provided that after decompression the data is exactly identical to the original.

(b) Color Configuration strictly identical: for works involving color presentation (e.g., digital images, video), the embedded ICC color profile, color space definition (e.g., sRGB, Adobe RGB, Display P3), color conversion tables, and any other color configuration information must be completely identical to the original, and must not be changed, removed, or replaced. If the original work had no embedded color configuration, it is deemed to use the default color space (typically sRGB), and no color configuration may be added when distributing.

Modification, addition, or deletion of metadata (EXIF, XMP, IPTC, file creation time, thumbnails, etc.) does not constitute modification of the Work and does not affect its status as an “Unmodified Artwork”, unless that metadata also serves as the carrier of Color Configuration (in which case the Color Configuration part is protected, while other parts are not).

1.4 “Impairment” means any act that results in any of the following:

    irreversible change to the Core Content Data (lossy compression, resampling, reduction of color depth, geometric simplification, audio resampling, etc.);

    change, removal, or incorrect replacement of the Color Configuration (including but not limited to stripping ICC profiles, forced color space conversion, ignoring embedded color configuration).

Other changes to metadata (non‑Color‑Configuration parts) do not constitute impairment. This definition takes precedence over any platform policy.

1.5 “Permissive Platform” means a network platform that does not apply lossy compression to uploaded Works, does not add watermarks, and does not restrict downloading of original file formats. (Metadata modification is no longer a disqualifying factor.)

1.6 “Encrypted Archive” means a distribution format created by encrypting and compressing a complete copy of the Work in accordance with the technical standards recommended in Article 6 of this License.

1.7 “Externally Forced Impairment” means impairment that an artist is compelled to apply to a Work due to mandatory requirements of laws, administrative regulations, judicial rulings, government directives, or platform terms of service (where the artist is not voluntary and cannot reasonably avoid). Voluntary technical choices by the artist (e.g., using lossy compression to save storage space) are not Externally Forced Impairment.

Article 6 – Technical Recommendation: Encrypted Archive Standard

To implement Article 3 of this License, the following technical standards for Encrypted Archives are recommended:

6.1 Encryption format

AES‑256 symmetric encryption is recommended, using compression software (e.g., 7‑Zip, WinRAR, PeaZip) to create an encrypted archive. The encryption password shall be distributed separately to recipients through a secure channel (e.g., encrypted instant messaging, end‑to‑end encrypted email).

6.2 Compression method

Only lossless compression algorithms are permitted (e.g., ZIP store mode, 7z using LZMA lossless compression). Any form of lossy processing (including but not limited to JPEG recompression, lossy audio encoding) shall not be applied to Works inside the Encrypted Archive.

6.3 Color Configuration and Metadata

This License mandates the preservation of the original Color Configuration (ICC profile, etc.). Other metadata may be freely handled. The Encrypted Archive shall ensure that Color Configuration information is not stripped by the platform or during transmission.

6.4 Obligations after decryption

Upon decryption of the Work, the recipient obtains authorization for the Work, but must continue to comply with all distribution channel restrictions of this License and shall not publish the decrypted Work on platforms excluded under Article 3.2.

Appendix – Technical Recommendations (Non‑binding)

Recommended implementation paths:

    Encrypted archive toolchain: Use 7‑Zip to create an encrypted .7z archive with AES‑256 encryption, compression level set to “Store” (no compression) or “Fast” (lossless), ensuring the Core Content Data remains unmodified. Metadata may be freely handled, but Color Configuration must be preserved.

    Color Configuration verification: When saving images, use lossless formats such as PNG or TIFF with embedded ICC profile. Avoid lossy formats like JPEG (even if ICC is retained, pixel data is impaired). Before distribution, you may use tools like exiftool -icc_profile to extract and compare the Color Configuration against the original.

    Permissive platform recommendation: Consider self‑hosted distribution nodes based on IPFS, cloud storage services that respect original file formats, or shared storage nodes maintained collectively by Trust Network members. When using a self‑hosted IPFS node, AES‑256 can be used for content encryption and ECC for key protection, with blockchain technology to ensure data integrity and authenticity.

    Trust Network member identification: Establish a shared list of member public keys among Trust Network members for verifying encrypted password distribution, and create a simple inter‑party dispute coordination mechanism.
